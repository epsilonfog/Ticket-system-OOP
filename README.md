سیستم پشتیبانی تیکت (Ticket Support System)

این مخزن شامل پیاده‌سازی اولیه‌ی یک سیستم ساده‌ی پشتیبانی تیکت است که برای یک تمرین طراحی شی‌گرا (Object-Oriented Design) ارائه شده است.

در این پروژه، سیستم به‌صورت مرحله‌به‌مرحله تحلیل و بازآرایی (Refactor) می‌شود و در این فرآیند از الگوهای طراحی (Design Patterns) استفاده خواهد شد.


## 1. شناسایی مسائل و الگوهای طراحی

### 1.1 رفتار وابسته به وضعیت

هر تیکت در طول عمر خودش می‌تونه در وضعیت‌های مختلفی قرار بگیره؛ مثل

NEW، ASSIGNED، IN_PROGRESS، RESOLVED و CLOSED.

با تغییر وضعیت تیکت، رفتار سیستم هم تغییر می‌کنه. به همین دلیل لازم بود این تغییر رفتار به شکل مناسبی مدیریت بشه.

الگوی انتخاب‌شده: State

دلیل انتخاب: چون امکان مدیریت رفتارهای متفاوت هر وضعیت و انتقال بین آن‌ها رو به‌صورت منظم فراهم می‌کنه.
### 1.2 روش‌های متفاوت پردازش تیکت

تیکت‌ها بسته به نوعشون (مثلاً BUG یا GENERAL) به شکل‌های متفاوتی پردازش می‌شن و منطق یکسانی ندارن.

الگوی انتخاب‌شده: Strategy

دلیل انتخاب: این الگو اجازه می‌ده روش‌های مختلف پردازش بدون تغییر در ساختار اصلی سیستم انتخاب و استفاده بشن.
### 1.3 نحوه‌ی ایجاد اشیاء

سیستم باید بر اساس نوع تیکت تصمیم بگیره که از کدوم روش پردازش استفاده کنه.

الگوی انتخاب‌شده: Factory

دلیل انتخاب: باعث می‌شه منطق ساخت اشیاء از منطق اصلی برنامه جدا بشه و کد خواناتر و قابل توسعه‌تر باشه.


## 2. نمودار کلاس (Class Diagram)

در طراحی جدید، سیستم از اجزای زیر تشکیل شده است:

    کلاس Ticket که نقش کلاس اصلی (Context) را دارد
    اینترفیس TicketState به‌همراه کلاس‌های مربوط به هر وضعیت
    اینترفیس HandlingStrategy و پیاده‌سازی‌های مختلف آن
    کلاس HandlingStrategyFactory
    یک Logger برای ثبت رویدادها و تغییرات سیستم

این طراحی باعث شده ساختار کد ماژولارتر بشه و مسئولیت هر بخش به‌صورت مشخص‌تری جدا بشه. در نتیجه، هم خوانایی کد بهتر شده و هم توسعه یا تغییر بخش‌های مختلف سیستم ساده‌تر خواهد بود.
